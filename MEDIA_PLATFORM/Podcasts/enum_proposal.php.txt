<?php


enum PodcastEpisodeStatus: string
{
    // -------------------------------------------------------------------------
    // Creative and Assembly Phase
    // -------------------------------------------------------------------------

    // Podcaste Episode record created — the starting point.
    // Set automatically by the Create Podcast Episode Wizard
    case created                        = 'created';

    // Working on the episode theme and concept.
    // Sometimes will know what the theme is, and so will skip this step
    // Set manually
    case working_on_theme               = 'working-on-theme';

    // Theme of the episode is known.
    // Now am Working on the script.
    // Set manually
    case writing_script                 = 'writing-script';

    // When the editing is done, and want to prepare the script for recording.
    // Need to proof-read, do final edits, and 
    // add the intro and outro (to the script).
    // Set manually
    case ready_to_finalize_the_script   = 'ready-to-finalize-the-script';

    // Script is ready for recording    
    // Set automatically by the Finalize Script Wizard
    case script_is_finalized            = 'script-is-finalized';

    // Script is locked
    // Populate episode number, scheduled_date, and website fields.
    // Set automatically by the "Prepare For Recordiing Wizard"
    case ready_to_record                = 'ready-to-record';

    // Episode is recorded. 
    // Audio file needs trimming and initial editing.
    // Sometimes do this step immediately after recording, and so will skip this step 
    // Set manually
    case raw_audio_file_needs_editing   = 'raw-audio-file-needs-editing';

    // Episode is ready for for publishing.
    // Set manually
    case ready_for_publishing           = 'ready-for-publishing';

    // Work on this episode is paused. Will get back to it later
    // Set manually
    case on_hold = 'on-hold';

    // Work on this episode is cancelled.
    // But, want to keep this for reference
    // Set manually
    case cancelled = 'cancelled';


    // -------------------------------------------------------------------------
    // Publishing Phase
    // 
    // The Publishing Wizard uses these statuses internally.
    // These statuses are set automatically by the Publishing Wizard only.
    // -------------------------------------------------------------------------

    // WAV audio file is ready to be uploaded to S3.
    // Set automatically
    // This is a long-time established step that might end up 
    // staying in this enum list, but is not used in the publishing wizard.
    case ready_to_upload_recording       = 'ready-to-upload-recording';

    // Raw recording uploaded — ready to submit to Auphonic.
    case ready_for_auphonic              = 'ready-for-auphonic';

    // Submitted to Auphonic — currently being processed.
    case processing_at_auphonic          = 'processing-at-auphonic';

    // Auphonic complete — awaiting review before next step.
    case auphonic_complete               = 'auphonic-complete';

    // Production audio ready — uploading to S3 and R2.
    case ready_to_upload_production_file = 'ready-to-upload-production-file';

    // Production audio file uploaded — ready to generate RSS feed.
    case ready_to_generate_rss_feed      = 'ready-to-generate-rss-feed';

    // RSS feed generated — ready to upload to S3 and R2.
    case ready_to_upload_rss_feed        = 'ready-to-upload-rss-feed';

    // RSS feed uploaded — ready to publish on website.
    case ready_to_publish                = 'ready-to-publish';

    // Live — permanent, public, locked.
    case published                       = 'published';

    // Live — permanent, not public, locked. And not published.
    case not_published                   = 'not-published';
}

Yes, this is really doing version 2 of the podcast part of my application. So, we need to think of the usual, broader issues going from version 1 to version 2.

Data Migration Risk:

There is no data migration risk. 

There is no live data in the "podcast_episodes_drafts" table. 

All records in the "podcast_episodes" tables have status = published. Except for one, which is the new status = not_published. 

I actually prefer the idea that there is a database table for each phase. One for the "Creative and Assembly" phase. And, one for the "Published" phase. Then, I know where all my published episodes are, it matches my RSS and websites. And, my application physically touches it as little as possible. 

Just leave the "podcast_episodes" table as it is. For the produciton/publication end of things, it is essentially all done. Nothing needs to touch it. 

I absolutely love the idea that there is a physical hand-off from one phase to the other, as represented by the two episode database tables. 

I love that the published data is physically as untouched as possible. And, that the records match something that exists in the real world. 

I love the idea that I can look at the published database table directly, and everything it has pretty much relates to publishing it. 

I love the idea that I can mess around with the creative-assembly database table, and none of it will imperil the published episodes. And, my fields can relate to the creative process + assembly process + planning process. I feel much less constrained. 

I love that I can have a completely separate mental model for each phase, and then not have to mentally reconcile, or mentally keep track, of which field represents which phase.

I absolutely love the idea that when an episode is published, it exists in the published database. And its corresponding record in the creative-assembly database can be deleted. Not a soft delete either. Physically deleted. So that the database stays lean. 

You said: "The website fields should persist — not be wiped." For the creative-assemply database table, it is fine to overwrite. I am consciously not worried about versioning. And, the permanent values for the published status are in another table. 

The API serves data from the published database table. 

Yes, we need to take the refactor to version 2 one step at a time.
