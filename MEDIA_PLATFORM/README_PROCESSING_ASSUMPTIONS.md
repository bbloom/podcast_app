# Processing Pipeline — Core Assumptions & Design Decisions

This document explains the fundamental assumptions behind the content processing
pipeline and how edge cases are handled. These decisions affect all three
processor classes:

- `YoutubeContentProcessor`
- `PodcastContentProcessor`
- `TextBasedRssContentProcessor`

**Read this before modifying any processor class.**

---

## 1. Feed Sort Order Assumption

**We assume all content sources sort their items newest-first.**

This is the foundation of the entire processing strategy. Feed files are
generated by algorithms, not by people, and are almost universally sorted in
reverse chronological order (newest item first). This assumption allows us to
walk a feed from the top and stop as soon as we reach content we have already
seen, rather than inspecting every item every time.

**This assumption is not guaranteed.** If a source violates it, some items may
be missed. We accept this trade-off. The alternative — inspecting every item on
every run — is expensive and unnecessary for well-behaved sources.

---

## 2. The Bookmark — `content_already_processed` Table

We maintain **one row per list_source** in the `content_already_processed`
table. This row is called the "bookmark". It stores the `source_url` of the
most recently processed item from that source.

The bookmark is the sole mechanism for preventing duplicate processing across
runs. It is completely independent of the `summaries` table. This means:

- You can safely delete rows from `summaries` after a digest is sent.
- Deleting summaries will **not** cause items to be reprocessed.
- The `summaries` table is for digest delivery only.

---

## 3. First Run

On the very first processing run for a list_source, **no bookmark exists**.

The processor detects this and applies a **lookback window**
(`config('processing.first_run_lookback_days')`). Only items published within
this window are processed. Items older than the window are skipped. This
prevents flooding the `summaries` table with back-catalogue content on the
first run.

After the first run completes, a bookmark is inserted pointing to the newest
item that was processed.

If no items fell within the lookback window, **no bookmark is inserted**. The
next run will also be treated as a first run.

---

## 4. Regular Runs (Bookmark Exists)

On every subsequent run, the processor walks the feed newest-first and checks
each item against the bookmark:

1. **Item URL matches the bookmark URL** → stop processing. This is the normal
   case. Trigger the bookmark rotation sequence (see section 5).

2. **Item is older than the bookmark's `processed_at` timestamp** → the
   bookmarked item has disappeared from the feed (e.g. a deleted YouTube video
   or a podcast episode that was pulled). Treat this as the stop signal and
   trigger the bookmark rotation sequence. Do not process this item or anything
   after it.

3. **Item URL is new** → process it normally. Continue to the next item.

---

## 5. Bookmark Rotation Sequence

When the stop signal is reached (cases 1 or 2 above), and **at least one item
was processed in the current run**, the bookmark is rotated:

1. `DELETE FROM content_already_processed WHERE list_source_id = ?`
2. `INSERT INTO content_already_processed (list_source_id, user_id, source_url, processed_at) VALUES (...)`

These are two separate operations, not an UPDATE. This is intentional — the
explicit two-step makes the intent obvious and avoids any ambiguity about what
the "current" bookmark is during the operation.

**If nothing was processed** (the very first item in the feed was already the
bookmark), the bookmark is left unchanged. No DELETE or INSERT is performed.

---

## 6. Edge Cases

### The bookmarked item has disappeared from the feed
Handled by comparing the item's `published_at` date against the bookmark's
`processed_at` timestamp. If the item being inspected is older than when the
bookmark was created, we have passed the point where the bookmarked item would
have appeared. Stop processing and rotate the bookmark if anything was processed.

### Out-of-order feed items
Not handled. If a source produces items in a non-standard order, some items
may be missed or duplicates may be processed. This is a known limitation,
documented here so future developers are aware. The assumption of sorted feeds
is fundamental to this design.

### Multiple list_sources for the same source
A single YouTube channel, podcast, or RSS feed can belong to many lists. Each
`list_source` has its own independent bookmark. Processing one list has no
effect on another list's bookmark, even if they point to the same underlying
source.

### First run produces no items within the lookback window
No bookmark is inserted. The next run is also treated as a first run. This
continues until at least one item falls within the lookback window.

---

## 7. What This Design Replaces

Previously, deduplication was performed by checking the `summaries` table:

```php
// OLD — do not use
$exists = DB::table('summaries')
    ->where('list_source_id', $listSource->id)
    ->where('source_url', $sourceUrl)
    ->exists();
```

This was fragile because `summaries` rows are deleted after digest delivery,
which caused previously-processed items to be reprocessed on the next run.

The `content_already_processed` bookmark table is purpose-built for this
concern and is never cleaned up as part of digest delivery.
